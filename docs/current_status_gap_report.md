# 文瀛餐厅管理系统现况与缺口分析报告

生成日期：2026-06-17

## 1. 结论摘要

当前仓库已经具备成熟的若依后台管理系统底座，也已经形成一套相对完整的“文瀛餐厅管理系统”施工方案文档，但业务功能尚未真正落地到代码层。

换句话说，当前状态不是“业务系统已完成待优化”，而是“底座可用、方案明确、业务模块待实施”。后续工作重点应放在将文档中的菜品、食材、库存、配方、订单、看板六类业务模块落到数据库、后端接口、前端页面、菜单权限和演示数据中。

## 2. 当前仓库结构

### 2.1 顶层结构

| 路径 | 当前作用 | 判断 |
|------|----------|------|
| `RuoYi-Vue/` | 主要代码仓库，包含若依后端、前端、SQL、文档副本 | 当前实际施工目录 |
| `docs/` | 项目需求、方案、备份资料 | 当前规划与说明资料目录 |
| `docs/plan_ruoyi_classic/` | 若依经典版施工方案 | 当前最贴近实际代码底座的有效方案 |
| `docs/backup/` | 旧方案归档 | 只作参考，不应作为施工依据 |

### 2.2 代码仓库状态

`F:\restaurant_manager` 顶层不是 Git 仓库，真正的 Git 仓库位于 `RuoYi-Vue/`。

`RuoYi-Vue/` 当前存在未跟踪文档目录：

- `doc/plan_ruoyi_classic/`
- `doc/require-unc/`

这说明若依代码仓库内的餐厅业务方案文档尚未纳入版本控制。若后续要以 `RuoYi-Vue/` 作为交付仓库，应先决定这些文档是否需要提交。

## 3. 实际技术底座

### 3.1 后端

实际后端为 RuoYi-Vue 经典版多模块 Maven 项目。

关键证据：

- `RuoYi-Vue/pom.xml` 中项目版本为 `3.9.2`，模块包括 `ruoyi-admin`、`ruoyi-framework`、`ruoyi-system`、`ruoyi-quartz`、`ruoyi-generator`、`ruoyi-common`。
- Java 版本配置为 `17`。
- Spring Boot 版本配置为 `4.0.6`。
- 后端启动入口为 `ruoyi-admin/src/main/java/com/ruoyi/RuoYiApplication.java`。

当前后端主要具备若依通用后台能力：

- 用户、角色、菜单、部门、岗位、字典、参数配置
- 登录、JWT Token、权限校验
- 操作日志、登录日志
- 在线用户、缓存监控、服务监控
- 定时任务
- 代码生成器

### 3.2 前端

实际前端为 Vue 2 + Vue CLI + Element UI。

关键证据：

- `RuoYi-Vue/ruoyi-ui/package.json` 中 `vue` 为 `2.6.12`。
- UI 组件库为 `element-ui` `2.15.14`。
- 前端脚本仍是若依默认的 `dev`、`build:prod`、`build:stage`。

当前前端主要具备若依通用后台页面：

- 登录页
- 首页仪表盘模板
- 系统管理
- 系统监控
- 系统工具
- 代码生成

### 3.3 数据库与外部依赖

实际数据库配置指向 MySQL。

关键证据：

- `RuoYi-Vue/ruoyi-admin/src/main/resources/application-druid.yml` 配置主库为 `jdbc:mysql://localhost:3306/ry-vue`。
- 默认账号为 `root`，默认密码为 `password`。
- `RuoYi-Vue/ruoyi-admin/src/main/resources/application.yml` 配置 Redis 地址为 `localhost:6379`。

## 4. 业务目标状态

当前有效业务目标来自 `plan_ruoyi_classic` 文档集，核心定位是高校食堂后台管理系统，不做真实点餐小程序和在线支付。

目标业务模块包括：

| 模块 | 文档目标 |
|------|----------|
| 菜品管理 | 分类、菜品、上下架、查询筛选 |
| 食材与库存 | 食材档案、自动初始化库存、入库、库存预警 |
| 菜品配方 | 菜品与食材用量关系，支持覆盖保存 |
| 订单管理 | 后台模拟下单、订单列表、详情、完成订单、退单 |
| 数据看板 | 今日营收、今日订单数、近 7 天趋势、销量 Top10、库存预警 |
| 权限角色 | 管理员、食堂经理、库管员三类角色与菜单隔离 |

核心演示闭环应为：

`配方维护 -> 食材入库 -> 模拟下单 -> 完成订单 -> 库存自动扣减 -> 看板统计展示`

### 5.2 文档能力

- 项目范围

- 角色职责

- 核心算法说明，包括配方扣库存、模拟下单、入库、库存预警和看板统计

## 6. 主要缺口

### 6.1 业务代码缺口

当前代码中未发现成型的餐厅业务模块实现。业务关键词主要出现在文档中，而不是后端 Controller、Service、Mapper 或前端页面中。

待实现内容包括：

- 菜品分类后端与前端 CRUD
- 菜品后端与前端 CRUD、上下架
- 食材档案后端与前端 CRUD
- 新增食材时自动初始化库存
- 入库记录与库存累加
- 配方查询与保存
- 模拟下单
- 完成订单并按配方扣库存
- 库存不足时事务回滚
- 退单逻辑
- 看板统计接口
- 看板前端页面

### 6.2 数据库缺口

`schema.sql` 已经提供业务表草案，但它目前仍是文档文件，不等于数据库已完成落库。

待确认或完成内容：

- 是否已经在本地 MySQL `ry-vue` 库执行过若依初始化 SQL
- 是否已经执行过餐厅业务 `schema.sql`
- 业务表是否需要补充审计字段，例如 `create_by`、`update_by`、`remark`，以更贴合若依生成器习惯
- 菜品、食材、订单等表是否需要统一软删除、状态字典和索引策略
- 是否需要把业务菜单、角色、权限初始化 SQL 纳入交付脚本

### 6.3 前端缺口

当前前端仍以若依默认页面为主，业务页面尚未出现。

待实现内容包括：

- 左侧业务菜单：餐厅管理、菜品管理、库存管理、配方管理、订单管理、数据看板
- 菜品、食材、库存、订单等列表页和弹窗
- 配方维护交互
- 模拟下单交互
- 完成订单按钮与库存扣减反馈
- 首页或独立看板替换若依默认介绍页
- 系统标题、Logo、版权等展示口径调整为“文瀛餐厅管理系统”

### 6.4 文档口径缺口

当前存在两套技术口径：

- `docs/README.md` 写的是 Vue 3 + TypeScript + Element Plus + Spring Boot 3 + MyBatis-Plus。
- `docs/plan_ruoyi_classic/README.md` 和实际代码对应的是 RuoYi-Vue 3.9.2 经典版、Vue 2、Element UI、MyBatis。

建议以后明确以 `plan_ruoyi_classic` 为唯一施工口径，并修正或降级根目录 `docs/README.md` 的技术栈说明，避免答辩或交接时产生误解。

### 6.5 版本控制缺口

`RuoYi-Vue/doc/plan_ruoyi_classic/` 与 `RuoYi-Vue/doc/require-unc/` 当前为未跟踪文件。核心算法文档已在顶层 `docs/plan_ruoyi_classic/` 与项目内 `RuoYi-Vue/doc/plan_ruoyi_classic/` 两处同步，但后续仍需明确文档主目录和版本控制策略。

建议后续处理方式：

- 如果 `RuoYi-Vue/` 是最终交付仓库，应提交这些文档。
- 如果顶层 `docs/` 才是文档主目录，则应避免在 `RuoYi-Vue/doc/` 保留重复副本，或者明确同步规则。

## 7. 风险判断

| 风险 | 当前表现 | 建议 |
|------|----------|------|
| 文档与代码漂移 | 根 README 技术栈与实际若依经典版不一致 | 先统一施工口径 |
| 业务范围膨胀 | 文档列出支付、小程序、供应商等不做项 | 严格守住不做范围 |
| 代码生成后改动分散 | 若直接生成多个模块，容易散乱 | 先统一包名、菜单、接口路径 |
| 扣库存逻辑复杂 | 需要事务、库存不足校验、配方计算 | 优先单独实现并测试订单完成逻辑 |
| 演示数据不足 | 看板依赖近 7 天订单数据 | 提前准备初始化/演示 SQL |
| 环境依赖未确认 | MySQL、Redis、Node、Maven 是否可用尚未验证 | 开工前先跑通若依原始系统 |

## 8. 推荐下一步施工顺序

### 第 0 步：统一口径与环境

目标：确认从哪套文档、哪套代码出发。

建议操作：

1. 明确 `RuoYi-Vue/` 为实际施工仓库。
2. 明确 `docs/plan_ruoyi_classic/` 为唯一有效施工方案。
3. 修正根目录 `docs/README.md` 中与实际技术栈不一致的描述。
4. 确认 MySQL、Redis、JDK、Maven、Node 环境可用。
5. 跑通原始若依后端和前端登录。

### 第 1 步：落库与菜单

目标：让若依识别业务表，并能生成基础代码。

建议操作：

1. 导入若依初始化 SQL。
2. 审核并执行餐厅业务 `schema.sql`。
3. 通过若依代码生成器导入业务表。
4. 生成基础 CRUD。
5. 配置业务菜单、角色、测试账号。

### 第 2 步：优先实现主闭环后端

目标：业务逻辑先正确，再做页面美化。

建议优先级：

1. 食材新增自动初始化库存。
2. 入库记录增加库存。
3. 配方查询与覆盖保存。
4. 模拟下单。
5. 完成订单并按配方扣库存。
6. 库存不足时阻止完成订单并回滚。

### 第 3 步：补前端业务页面

目标：能在浏览器中完整演示。

建议顺序：

1. 菜品分类与菜品页面。
2. 食材与库存页面。
3. 配方维护页面。
4. 订单列表、详情、模拟下单、完成订单。
5. 数据看板。

### 第 4 步：演示数据与验收

目标：形成可答辩版本。

建议操作：

1. 准备菜品、食材、配方初始化数据。
2. 准备近 7 天订单演示数据。
3. 按三角色验证菜单权限。
4. 按主链路完整彩排。
5. 补充安装说明、用户说明、演示脚本。

## 9. 建议的最小可交付版本

如果时间紧，建议只保住以下能力：

1. 管理员能登录并配置用户、角色、菜单。
2. 食堂经理能维护菜品、配方、订单、看板。
3. 库管员能维护食材、入库、库存。
4. 能完成一次真实演示链路：维护配方 -> 入库 -> 模拟下单 -> 完成订单 -> 库存扣减 -> 看板变化。
5. 库存不足时系统能明确提示，并且不会错误扣减。

这个版本足够支撑“高校食堂后台管理系统”的答辩叙事，也能体现系统不是简单 CRUD，而是有业务联动闭环。

## 10. 我建议的接手方式

我建议下一轮直接进入“第 0 步 + 第 1 步”：先统一文档口径、验证环境、审查 `schema.sql` 是否适合若依代码生成器，然后再开始生成并接入第一批业务模块。

具体来说，我可以继续做三件事：

1. 修订文档口径，把根目录 README 与若依经典版方案对齐。
2. 审查并调整业务表结构，使其更适合若依生成器和后续业务逻辑。
3. 制定第一批实现清单，按菜品、食材、库存、配方、订单、看板逐步落地。
