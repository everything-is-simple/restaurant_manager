# 文瀛餐厅管理系统 — 3-5 分钟答辩演示脚本

## 1. 演示目标

用 3-5 分钟说明系统解决了高校食堂后台管理中的三个问题：菜品与配方统一维护、库存随业务自动联动、经营数据实时汇总。

核心亮点一句话：完成订单时，系统会按菜品配方自动扣减食材库存，并用事务和 `UPDATE ... WHERE stock >= deduct` 防止库存超扣。

## 2. 演示前准备

- 后端已启动：`http://127.0.0.1:8080`
- 前端已启动：`http://127.0.0.1`
- 使用账号：`manager/admin123`、`stockkeeper/admin123`
- D10 演示数据已存在：56 道菜品、36 种食材、110 条配方、212 单已完成订单、416 条订单明细。

## 4. 关键讲解点

### 4.1 为什么要有配方

菜品本身只记录名称、分类、价格和上下架状态，不能直接知道需要扣哪些食材。配方表把菜品和食材建立多对多关系，保存每份菜需要的食材用量。

### 4.2 为什么完成订单才统计营收

模拟下单只是生成待处理订单，只有完成订单后才代表真实出餐，所以看板的今日营收、今日订单数和趋势都以“已完成订单”的 `complete_time` 为口径。

### 4.3 库存不足怎么处理

扣库存 SQL 使用 `stock >= deduct` 条件。如果库存不足，更新行数为 0，服务层抛出业务异常，事务回滚，订单状态和库存都不会被写脏。

### 4.4 权限隔离怎么体现

食堂经理负责经营侧：菜品、配方、订单、看板；库管员负责库存侧：食材、入库、库存。菜单和按钮权限由角色控制，避免跨职责操作。

## 5. 常见追问回答

| 问题 | 回答 |
|------|------|
| 为什么不做点餐小程序？ | 本项目范围是后台管理系统，点餐端用模拟下单替代，重点展示后台闭环和库存扣减算法。 |
| 如何避免前端篡改菜名和价格？ | 下单时后端只信任 dishId，菜名和价格从数据库菜品表读取。 |
| 如何避免库存扣成负数？ | 扣减 SQL 带 `stock >= deduct` 条件，并且整个完成订单过程在事务中执行。 |
| 为什么 TopN 按 dishId 聚合？ | 避免历史订单明细中的菜名脏数据导致同一道菜被拆分统计。 |
| 演示数据是否真实？ | D10 演示数据按文瀛餐厅午餐约 1000 人规模估算，覆盖 56 道菜、36 种食材和近 7 天订单。 |

## 6. 收尾话术

本系统完成了餐厅后台从基础资料、库存入库、配方维护、订单处理到经营统计的完整闭环。答辩重点不是页面数量，而是完成订单后能按配方自动扣库存，并且在库存不足时可靠回滚，这体现了业务建模、事务控制和数据统计的一致性。
