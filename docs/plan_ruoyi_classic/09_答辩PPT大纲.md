# 文瀛餐厅管理系统 — 答辩 PPT 大纲

## 第 1 页：标题页

- 标题：文瀛餐厅管理系统
- 副标题：高校食堂后台管理与库存联动系统
- 项目类型：课程/答辩项目
- 技术路线：RuoYi-Vue 3.9.2 + Spring Boot + Vue 2 + MySQL

讲稿：本项目面向高校食堂后台管理，目标是跑通菜品、库存、订单和经营看板的核心业务闭环。

## 第 2 页：项目背景与问题

- 菜品、食材、库存信息分散，维护成本高。
- 食材损耗和库存预警依赖人工盘点。
- 订单数据与经营统计不统一。
- 答辩需要一个可运行、可演示、能解释核心算法的系统。

讲稿：食堂后台最核心的问题不是单独录入数据，而是业务发生后数据能自动联动。

## 第 3 页：需求范围

- 做：分类、菜品、食材、库存、入库、配方、订单、看板。
- 不做：点餐小程序、在线支付、供应商管理。
- 演示闭环：配方维护 -> 食材入库 -> 模拟下单 -> 完成订单 -> 扣库存 -> 看板统计。

讲稿：为了控制课程项目范围，前台点餐用后台模拟下单替代，重点放在后台链路和库存算法。

## 第 4 页：系统架构

- 前端：Vue 2、Element UI、ECharts。
- 后端：Spring Boot、Spring Security、JWT、MyBatis XML。
- 数据层：MySQL 8、Redis。
- 权限：菜单权限 + 按钮权限 + 后端注解。

讲稿：系统沿用成熟后台框架，但餐厅业务模块、扣库存逻辑、配方维护和看板统计都是围绕本项目补充实现。

## 第 5 页：数据模型

- 基础表：分类、菜品、食材、库存。
- 业务表：入库、配方、订单、订单明细。
- 核心关系：菜品通过配方关联多种食材；订单明细通过菜品反查配方。

讲稿：配方表是整个系统的连接点，订单完成时不是扣菜品库存，而是扣菜品背后的食材库存。

## 第 6 页：核心业务闭环

- 菜品维护：分类、售价、上下架。
- 配方维护：每道菜配置食材和每份用量。
- 入库登记：新增、修改、删除入库记录联动库存。
- 订单处理：模拟下单、完成订单、退单回滚。
- 看板统计：营收、订单、销量、预警。

讲稿：演示时会从配方开始，经过下单和完成订单，最后在库存和看板上看到真实变化。

## 第 7 页：扣库存算法亮点

- 完成订单使用事务控制。
- 扣减数量 = 菜品份数 × 配方中每份食材用量。
- 扣库存 SQL 使用 `UPDATE ... WHERE stock >= deduct`。
- 库存不足时抛出异常，订单状态和库存整体回滚。

讲稿：这是项目最值得讲的技术点，解决了库存不能扣成负数、订单不能半成功的问题。

## 第 8 页：权限设计

- admin：全部系统和餐厅功能。
- MANAGER：菜品、配方、订单、库存查看、数据看板。
- STOCK_KEEPER：食材、入库、库存。
- 权限同步：后端 `@PreAuthorize`、前端按钮、菜单 SQL 三处对齐。

讲稿：系统按食堂岗位分工做了角色隔离，避免经理越权维护食材，库管越权处理订单。

## 第 9 页：演示数据

- 56 道菜品。
- 36 种食材。
- 110 条配方。
- 212 单已完成订单。
- 416 条订单明细。
- 日期覆盖 2026-06-14 至 2026-06-20。

讲稿：演示数据不是空页面凑数，而是为看板趋势、销量 Top10 和库存预警准备的可解释数据。

## 第 10 页：页面效果

- 首页：展示系统定位和主流程。
- 订单页：模拟下单、订单详情、完成、退单。
- 配方页：按菜品维护食材用量。
- 看板页：4 个指标卡、近 7 天趋势、销量 Top10、库存预警。

讲稿：这一页配合截图展示，证明系统不是只有接口，而是有完整可操作页面。

## 第 11 页：测试与质量

- 后端编译：`mvn -pl ruoyi-admin -am compile`。
- 前端构建：`npm run build:prod`。
- 接口验证：订单、配方、库存、看板核心接口。
- 页面验证：manager 与 stockkeeper 两角色菜单和按钮。
- 主链验证：模拟下单 -> 完成订单 -> 库存扣减 -> 看板变化。

讲稿：系统经过代码审查、接口验证、页面回归和主链彩排，答辩时优先演示已经验证过的稳定路径。

## 第 12 页：总结与展望

- 已完成：餐厅后台核心闭环、角色权限、演示数据和经营看板。
- 项目价值：把食堂日常经营数据从人工维护推进到自动联动。
- 后续可扩展：点餐小程序、供应商管理、采购审核、移动端看板。

讲稿：当前版本适合课程答辩和后台闭环演示，后续如果扩展真实业务，可以继续接入前台点餐和采购流程。
