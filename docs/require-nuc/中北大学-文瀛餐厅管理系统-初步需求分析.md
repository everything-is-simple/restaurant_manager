# 中北大学-文瀛餐厅管理系统-初步需求分析

## 一、学校原始要求再确认

根据项目详情页，学校给出的关键信息如下：

- 项目名称：文瀛餐厅管理系统
- 所属院系：计算机科学与技术学院（大数据学院）
- 指导老师：王斌
- 实施平台：自主可控创新实验室
- 项目来源：其他
- 验收方式：答辩 + 软件
- 参考专业：计算机科学与技术
- 参与学生数量上限：3

项目简介中的关键词是：

- 高校食堂后台管理系统
- 菜品上架
- 库存盘点
- 订单统计
- 就餐数据分析
- 联动前端点餐数据
- 管控食材损耗
- 优化菜品供给
- 实现食堂精细化运营

## 二、学校要求的真实含义

这个项目不是要做一个完整商业化外卖平台，也不是要做一个移动端点餐系统主体。学校要求的重点，其实是一个“可运行的食堂后台管理系统”，并且要能说明它如何帮助食堂做精细化运营。

换句话说，老师真正会看的是三件事：

1. 软件能不能跑起来，有没有实际界面和业务流程
2. 功能是不是围绕食堂后台，而不是做偏成商城或支付系统
3. 有没有一个足够清晰的技术亮点来支撑答辩

## 三、当前方案与学校要求的一一对应

### 1. “高校食堂后台管理系统” 对应什么

我们当前采用若依经典版作为底座，做的是 PC 端后台管理系统，具备登录、权限、菜单、数据管理和业务页面，完全符合“后台管理系统”的要求。

对应模块：

- 登录与权限管理
- 菜品管理
- 食材管理
- 库存管理
- 订单管理
- 数据看板

### 2. “菜品上架” 对应什么

项目要求里明确提到了菜品上架，因此菜品模块必须保留。

当前方案对应：

- 菜品分类管理
- 菜品新增、编辑、删除
- 菜品上架 / 下架状态切换
- 菜品列表筛选与查询

这部分可以直接在答辩中展示为：系统已经支持食堂菜品的基础运营维护。

### 3. “库存盘点” 对应什么

学校要求不是简单地记菜品数量，而是要体现食材层面的库存管理。

当前方案对应：

- 食材档案管理
- 食材入库登记
- 当前库存量查询
- 低库存预警

这部分说明系统不是停留在菜单展示，而是真正管理食堂后端物料。

### 4. “订单统计” 对应什么

学校明确提到订单统计，所以必须有订单来源和订单汇总能力。

当前方案对应：

- 后台模拟下单
- 订单列表
- 订单详情
- 今日订单数统计
- 今日营收统计

这里的“模拟下单”不是偷工减料，而是为了在没有真实前端点餐系统接入的情况下，依然能完成订单数据联动和后续统计验证。

### 5. “就餐数据分析” 对应什么

学校要求里明确写了就餐数据分析模块，因此看板必须保留。

当前方案对应：

- 今日营收
- 今日订单数
- 近 7 天营收趋势
- 菜品销量 Top10
- 库存预警列表

这部分是答辩时最直观的页面，用来体现系统不仅能记录数据，还能做基础分析。

### 6. “联动前端点餐数据” 对应什么

这句话最容易被误解。它不一定要求本项目必须开发一个完整前端点餐小程序，而是要求后台系统具备承接和处理点餐数据的能力。

当前方案对应：

- 通过订单模块接收点餐结果
- 当前阶段使用“后台模拟下单”替代真实前端接入
- 预留标准订单接口，后续可以由小程序或网页点餐端接入

答辩时建议这样表达：

“本项目当前聚焦食堂后台管理端。为了在有限周期内优先完成后台业务闭环，我们使用模拟下单方式验证订单流转与库存联动，同时保留标准订单接口，后续可无缝接入前端点餐系统。”

### 7. “管控食材损耗、优化菜品供给” 对应什么

这是整个项目最关键的业务亮点，也是最值得讲的技术点。

当前方案对应：

- 菜品与食材通过配方表关联
- 完成订单后按配方自动扣减库存
- 通过库存变化反映理论消耗
- 通过销量和库存预警辅助食堂调整供给

这说明系统不是简单的 CRUD，而是围绕食堂运营逻辑建立了“菜品 -> 配方 -> 食材 -> 库存 -> 订单 -> 看板”的闭环。

## 四、当前方案为什么没有偏题

当前方案保留的内容，正好覆盖学校要求里的全部主关键词：

- 菜品上架：有
- 库存盘点：有
- 订单统计：有
- 就餐数据分析：有
- 联动点餐数据：有，用模拟下单完成现阶段验证
- 食材损耗管控：有，通过配方扣库存实现
- 精细化运营：有，通过订单、库存、看板联动体现

同时，当前方案主动砍掉了这些不在学校核心要求里的内容：

- 在线支付
- 外卖配送
- 会员系统
- 小程序前台主体
- Docker 部署
- 复杂供应链

这不是缩水，而是为了把项目准确收敛在“学校真的要什么”上。

## 五、答辩时最推荐的表达

可以直接用下面这段作为核心说明：

“文瀛餐厅管理系统的定位不是一个完整的商业点餐平台，而是一个面向高校食堂的后台管理系统。系统围绕菜品上架、库存盘点、订单统计和就餐数据分析四类核心需求展开，并通过菜品配方与食材库存联动，实现订单完成后自动扣减库存，从而帮助食堂识别食材消耗情况、优化菜品供给，体现精细化运营能力。当前版本聚焦后台闭环，前端点餐数据通过模拟下单方式接入，后续可进一步对接真实点餐系统。” 

## 六、结论

学校要的不是“大而全”，而是“方向对、系统能跑、业务闭环清楚、答辩能讲透”。

因此，当前收束后的若依经典版方案是合适的，重点应该继续放在：

- 跑通环境
- 做完主业务闭环
- 把配方扣库存讲清楚
- 用看板把“精细化运营”展示出来
